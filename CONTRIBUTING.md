# Contributing to This Repository

## General Guidance for Clients

Shared Services Canada (SSC) welcomes contributions and suggestions to this project via GitHub issue reports.

## Before Submitting a Pull Request (PR)

- **Communicate with SSC:** If a resource has been assigned to a ticket, contact SSC via email before submitting a PR for that ticket. If no resource is assigned, you can fork the repository and submit a PR for collaboration.
- **License:** This repository uses the MIT License. All contributions are licensed under MIT.
- **Branch management:** Direct commits to the `main` branch are not allowed. Contributions must be submitted by PR from a forked repository. Each PR is reviewed by two SSC reviewers.

## Issue Submission

1. Search existing issues to avoid duplicates.
2. Use the appropriate issue template, if available. If not, submit a blank issue.
3. Provide all details requested in the template.
4. Link any related tickets, PRs, or documentation.

## Issue Triage and Collaboration

- **Review & labelling:** The Cloud Security Compliance Division (CSCD) reviews and labels new issues, confirms reproduction, and prioritizes items for the next release plan.
- **Duplicates:** Duplicates are linked to the original issue and closed.
- **Information requests:** CSCD may request additional information.
- **Resolution:** When a fix or change is merged, CSCD updates the issue, references the PR, and closes the issue.

## Pull Request (PR) Submission

1. Open a PR from your fork to the `main` branch.
2. Reference the related ticket/issue in the PR title or description.
3. Submit an issue/bug report first if one does not already exist.
4. Expect review by two SSC reviewers and address all review comments.

## PR Approval

- **Collaboration:** Collaboration between the client and the SSC Cloud Security Compliance Division (CSCD) is strongly encouraged.
- **Leadership and approval:** SSC CSCD leads integration, development, testing, and final approval to ensure SSC standards are met. Approval requires two SSC reviews and all required checks to pass.

## Collaboration Process

- **Assignment of resources:**
  - If an SSC CSCD resource is actively working on an issue in a branch, they will collaborate with the client according to the next release plan.
  - If no resource is assigned, SSC CSCD validates the PR against the Compliance as Code (CaC) design requirements. Based on the results, CSCD either assigns a resource to collaborate (per the next release plan) or closes the PR and communicates the decision.

- **Incorporation of changes:**
  - The assigned SSC CSCD resource may incorporate changes into the existing SSC CSCD branch or create a new SSC CSCD branch.
  - All changes undergo the required testing.
  - CSCD raises a new PR from the SSC CSCD branch, referencing the original client PR.

- **Communication:**
  - CSCD communicates the outcome to the client in the PR and/or linked issue.
  - The client’s original PR is closed with an explanatory comment.

## Security

Do not post any security issues on the public repository.

For security-related issues, follow the guidance in the [SECURITY.md](https://github.com/ssc-spc-ccoe-cei/azure-guardrails-solution-accelerator/blob/main/SECURITY.md) file.

# Contribuer à ce référentiel – Directives et attentes

## Directives générales pour les clients qui contribuent au référentiel

Services partagés Canada (SPC) invite les clients à apporter des contributions et à formuler des suggestions pour ce projet au moyen de rapports d’incident GitHub.

## Avant de soumettre une demande de tirage (« pull request » ou PR)

- **Communication avec SPC** : Si une ressource a été assignée à un billet, les clients doivent communiquer avec SPC par courriel avant de soumettre une PR pour ce billet. Si aucune ressource n’est assignée, les clients peuvent dupliquer le référentiel (« fork the repository ») et soumettre une PR pour collaborer.
- **Licence MIT** : Ce référentiel utilise la licence MIT. Toutes les contributions sont accordées en vertu de la licence MIT.
- **Gestion des branches** : Les validations directes sur la branche principale ne sont pas autorisées. Les contributions doivent être soumises au moyen d’une PR à partir d’un référentiel dupliqué (« forked repository »). Chaque PR est examinée par deux examinateurs de SPC.

## Étapes à suivre pour soumettre un incident

1. Effectuer une recherche sur les incidents/problèmes existants afin d’éviter les doublons.
2. Utiliser le modèle d’incident approprié, s’il est disponible. S’il n’est pas disponible, soumettre un incident vierge.
3. Fournir tous les renseignements demandés dans le modèle.
4. Lier les billets, PR ou documents connexes.

## Tri des incidents et collaboration

- **Examen et étiquetage** : La Division de la conformité à la sécurité infonuagique (DCSI) examine les nouveaux incidents, leur attribue une étiquette, confirme la reproduction et établit l’ordre de priorité des éléments pour le prochain plan de mise en production.
- **Incidents en double** : Les incidents en double sont liés à l’incident original, puis fermés.
- **Demandes d’information** : La DCSI peut demander des renseignements supplémentaires au besoin.
- **Résolution** : Lorsqu’un correctif ou un changement est fusionné, la DCSI met à jour l’incident, indique une référence à la PR et ferme l’incident.

## Étapes pour la soumission d’une demande de tirage (« pull request » ou PR)

- Ouvrir une PR à partir d’un fork (référentiel dupliqué) vers la branche principale.
- Indiquer le billet ou l’incident lié dans le titre ou la description de la PR.
  - Soumettre d’abord un rapport d’incident ou de bogue si un tel rapport n’existe pas déjà.
- S’attendre à ce que deux examinateurs de SPC vérifient la PR et traiter tous les commentaires d’examen.

## Étapes pour l’approbation d’une demande de tirage (PR)

- **Collaboration** : La collaboration entre le client et la Division de la conformité à la sécurité infonuagique de SPC est fortement encouragée.
- **Responsabilités de la DCSI** : Pour veiller au respect des normes de SPC, la DCSI dirige l’intégration, le développement, les tests et l’approbation finale. L’approbation nécessite deux examens de SPC et la réussite de toutes les vérifications obligatoires.

## Processus de collaboration

### Affectation des ressources

- Si une ressource de la DCSI de SPC travaille activement sur un incident dans une branche, elle collaborera avec le client conformément au prochain plan de mise en production.
- Si aucune ressource n’a été affectée, la DCSI de SPC valide la PR en fonction des exigences de conception de la Conformité en tant que code (CC). Selon les résultats, la DCSI affecte une ressource pour collaborer (selon le prochain plan de mise en production) ou ferme la PR et communique la décision.

### Intégration des changements

- La ressource de la DCSI de SPC peut intégrer les changements dans la branche actuelle de la Division ou créer une nouvelle branche de la DCSI de SPC.
- Tous les changements font l’objet des tests nécessaires.
- La DCSI prépare une nouvelle PR à partir de la branche de la DCSI de SPC et y fait référence à la PR originale du client.

### Communication

- La DCSI communique le résultat au client dans la PR et/ou dans l’incident lié.
- La PR originale du client est fermée avec une explication dans le commentaire.

## Sécurité

Ne publiez aucun incident lié à la sécurité dans le référentiel public.

Pour les questions liées à la sécurité, veuillez consulter le document [SECURITY.md](https://github.com/ssc-spc-ccoe-cei/azure-guardrails-solution-accelerator/blob/main/SECURITY.md).
